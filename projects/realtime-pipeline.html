Real-Time Event Pipeline

2024 Fintech Platform 5B+ events/day Sub-100ms latency
Project Architecture Diagram · Replace with actual image

This project involved designing and building a high-throughput real-time event streaming pipeline for a Series C fintech company processing payment events, user actions, and fraud signals across 40+ markets. The core constraint: every event had to be processed and available for downstream queries within 100 milliseconds at p99.

The Challenge

The existing batch-based system ran on 6-hour cycles, meaning fraud detection models worked with stale features and business teams couldn't react to intraday patterns. The engineering challenge was rebuilding this to stream without disrupting the existing fraud models that the business depended on.

Architecture Decisions

We chose Kafka as the event backbone for its durability guarantees and broad ecosystem support. Apache Flink handled the stateful stream processing — session windowing and late-event handling being the most complex pieces. Events flow from Kafka through Flink into PostgreSQL for the operational store and into our data warehouse for analytics.

Key Engineering Challenges

  • Exactly-once semantics across Kafka and PostgreSQL under partial failures
  • Backpressure management during traffic spikes (Black Friday saw 8× normal load)
  • Schema evolution without downtime across 12 event types
  • Late-arriving events reconciliation across timezone boundaries
  • Multi-region replication with active-passive failover under 30s

Results

The pipeline has been running in production for 14 months. P99 end-to-end latency sits at 68ms. We've had zero data loss incidents. The fraud team reported a 22% improvement in detection rates within 6 weeks of migration, attributable directly to fresher features.

Infrastructure costs actually decreased 18% compared to the old batch cluster, primarily because Flink's streaming model eliminated the need to re-process historical data on each run.
